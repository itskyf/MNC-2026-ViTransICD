# ON-2: Normalize Ontology from PDF and Build Bilingual Index

## Objective

Implement a silver-stage ontology normalizer that reads the bronze ICD-10 PDF page records from ON-1 and emits canonical 3-character ICD entries as schema-valid `OntologyCode` records.

The normalizer must produce a bilingual searchable ontology index using the official Vietnamese titles, extractable English titles, and normalized aliases.

## Dependency

This task depends on ON-1.

Use only this upstream artifact as input:

* `data/bronze/icd10_official_pdf/primary/document_records.jsonl`

Treat ON-1 output as the single source of truth for this task.

## Scope

Implement only 3-character ontology normalization and bilingual indexing from the official PDF text already parsed by ON-1.

Emit one canonical `OntologyCode` per official 3-character code.

Populate bilingual search features through:

* `title_vi`
* `title_en`
* `aliases`
* `search_text`

Write outputs only under:

* `data/silver/icd10_official_pdf/normalized_ontology/`

## Out of Scope

Do not emit 4-character codes.

Do not emit chapter, block, or tree artifacts in a custom schema.

Do not emit `DocumentRecord`, `MentionRecord`, `CandidateLink`, `SilverRecord`, `PredictionRecord`, or `ExplanationRecord`.

Do not add fields beyond `OntologyCode`.

Do not implement separate vector indexes, search engines, or database storage.

Do not create a new data layer outside `data/{bronze,silver,gold}/...`.

## Required Schema

Use only models already defined in `src/mnc/schemas/`.

For this task, the output artifact type is `OntologyCode` from `src/mnc/schemas/ontology.py`.

This task intentionally uses `OntologyCode` as the only persisted ontology artifact.

## Input Contract

Read `DocumentRecord` rows from:

* `data/bronze/icd10_official_pdf/primary/document_records.jsonl`

Required assumptions about input:

* records are page-level
* page order is preserved
* `raw_text` contains source-faithful page text
* `source` is `icd10_official_pdf`

The normalizer must fail loudly if the input file is missing, invalid, empty, or no 3-character codes are extracted.

## Output Contract

Emit one `OntologyCode` per official 3-character ICD code.

Write records to:

* `data/silver/icd10_official_pdf/normalized_ontology/ontology_codes.jsonl`

Parquet output is optional. If implemented, write to:

* `data/silver/icd10_official_pdf/normalized_ontology/ontology_codes.parquet`

Each output row must validate against `OntologyCode`.

## Record Granularity

Use 3-character code granularity.

Emit exactly one canonical record for each extracted official 3-character code.

Do not duplicate a code across multiple pages.

If the same code is encountered multiple times, merge evidence deterministically and keep one final record.

## Canonicalization Rules

Normalize `code_3char` as uppercase 3-character ICD code without dot.

Use deterministic parsing only.

Treat the code line nearest to the title as the canonical source line.

Use chapter context from the nearest preceding chapter heading.

Populate `title_vi` from the official Vietnamese title.

Populate `title_en` only when extractable with high confidence from the PDF text. Otherwise set `None`.

Do not hallucinate English titles.

## Alias Rules

Build `aliases` as normalized string variants derived only from the official PDF text.

Allowed alias sources:

* Vietnamese title normalization
* English title normalization when `title_en` exists
* parenthetical descriptors that belong to the same 3-character entry
* explicit synonym-like title variants on the same entry line

Alias rules:

* trim whitespace
* collapse repeated spaces
* lowercase for normalized matching
* preserve diacritics
* remove duplicates
* remove empty strings
* do not include the code itself as an alias
* do not inject external synonyms

## Search Text Rules

Populate `search_text` as a compact lexical index string for retrieval.

Include, in deterministic order:

* `code_3char`
* `title_vi`
* `title_en` if present
* unique aliases

Use a single separator strategy consistently across all rows.

Keep `search_text` non-empty and reproducible for the same input.

## Chapter Mapping Rules

Populate `chapter_id` from the nearest valid chapter heading that governs the 3-character entry.

If chapter resolution is impossible for an entry, set `chapter_id = None`.

Do not invent chapter IDs.

Keep chapter assignment deterministic.

## Extraction Rules

Use page-order parsing over ON-1 records.

Use explicit regex and layout heuristics to identify:

* chapter headings
* 3-character code lines
* bilingual titles on the same line or adjacent lines

Prefer simple and auditable heuristics over broad fuzzy parsing.

Reject malformed candidate codes that do not match the accepted 3-character pattern.

## Accepted Code Pattern

Use a strict 3-character ICD pattern appropriate for canonical labels.

Minimum accepted form:

* one uppercase letter
* two digits

Recommended regex:

* `^[A-Z][0-9]{2}$`

## Storage Layout

Use only:

* `data/silver/icd10_official_pdf/normalized_ontology/ontology_codes.jsonl`
* `data/silver/icd10_official_pdf/normalized_ontology/ontology_codes.parquet` if implemented

Do not create extra storage layers or custom ontology files outside this scope.

## Suggested Module Structure

Keep implementation small and task-focused.

Recommended pieces:

* bronze reader
* chapter state tracker
* 3-character entry parser
* bilingual title resolver
* alias normalizer
* `OntologyCode` builder
* JSONL writer
* optional Parquet writer
* CLI entrypoint

## Validation

Validate every output record with `OntologyCode` before writing.

Minimum validation checks:

* at least one record is produced
* all `code_3char` values are unique
* every `code_3char` matches the accepted 3-character pattern
* `title_vi` is non-empty for every record
* `search_text` is non-empty for every record
* `aliases` contains no duplicates
* `aliases` contains no empty strings
* `source` metadata is not added outside the schema
* output ordering is deterministic, preferably sorted by `code_3char`

## Logging

Log at least:

* input path
* number of bronze pages read
* number of 3-character entries extracted
* number of entries with `title_en`
* number of entries with non-empty aliases
* output path

Do not persist custom logs as data artifacts.

## Testing

Use `pytest`.

Keep tests simple and focused.

Required unit tests:

* schema validation test for one generated `OntologyCode`
* 3-character code normalization test
* duplicate-code merge or rejection test
* chapter propagation test on mocked page snippets
* bilingual title extraction smoke test
* alias deduplication test
* `search_text` composition test
* JSONL writer test
* parser smoke test on mocked ON-1 page records
* deterministic output ordering test for repeated runs on the same input

Do not require network access in unit tests.

Use small mocked `DocumentRecord` inputs instead of the real PDF.

## Acceptance Criteria

The task is complete when all conditions below are satisfied:

* normalizer reads only ON-1 bronze page records
* output is written only under `data/silver/icd10_official_pdf/normalized_ontology/`
* output records are valid `OntologyCode` instances
* one canonical record is emitted per extracted 3-character code
* `title_vi` is populated for all records
* `title_en` is populated only when directly supported by PDF text
* `aliases` and `search_text` provide a usable bilingual lexical index
* no custom ontology schema is introduced
* no 4-character outputs are mixed into this task
* `pytest` unit tests exist and pass
